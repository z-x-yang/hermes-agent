# Live–Fake Contract Parity Acceptance

Read this when a delegated implementation is green locally but production depends on external gateways, generated SQL, producer artifacts, or protected/live execution that has not yet run. The goal is to distinguish **tested behavior** from a fake-only universe.

## 1. Test doubles must implement the production protocol

A fake should differ in transport, not in return shape or semantics.

Audit the real adapter first, then require the fake to match:

- same method names and argument meanings;
- same row/container shapes (`list[dict]` vs tuple rows vs special summary dicts);
- same nullability, types, ordering guarantees, duplicate behavior, and exception classes;
- same lifecycle (materialize → query → cleanup, upload → verify, etc.).

Red flags:

```python
if label == "idf":
    return {"n": 100, "counts": {...}}  # production always returns tabular rows
```

```python
fake_events = [(person, code, date)]      # production returns dict-like rows
```

A label-specific convenience shape is acceptable only if the production adapter performs that exact conversion too. Otherwise the test validates a protocol that does not exist.

Behavioral gate: pass production-shaped rows through the full entry point, not just a parser helper. Include malformed/missing/duplicate/nonfinite rows and assert failure at the boundary.

## 2. Canonical builders must be invoked, not imitated

Static assertions such as “source contains `cohort_sql`” do not prove the approved builder is used. A module can redefine the same name or copy a weaker query.

Audit:

1. Resolve the imported callable/object identity.
2. Trace the real entry point to the call site.
3. Inspect the emitted query/config, not only source tokens.
4. Assert load-bearing semantics: joins/aliases are defined, denominator and time boundaries are exact, required filters are present, and staging inputs are materialized rather than assumed.
5. If a canonical builder exists, reuse it; add a thin staging adapter instead of duplicating its logic.

A green parser/unit suite cannot close SQL correctness when no test examines the generated query or executes a dry-run/validator against the target dialect.

## 3. Consumer fixtures should come from the real producer layout

When one phase consumes another phase's handoff, handwritten fixtures often drift into an easier format.

Acceptance fixture should be generated by the producer or copied from its exact documented layout and include:

- filenames/directories;
- manifest schema/version;
- checksums/fingerprints;
- ordering/alignment rules;
- typed identity and semantic metadata;
- PASS/FAIL summary or readiness gate.

Then run the consumer through that fixture. Also test one checksum mismatch, one missing file, one ordering mismatch, and one producer FAIL status. An unused required CLI path is a blocker, even if parsing helpers pass independently.

## 4. Green tests must cover the production boundary and complete matrix

For every normative requirement, map:

```text
requirement → production call site → behavioral test → emitted artifact/status
```

Look for omissions hidden by aggregation:

- only three of four `(window × view)` cells execute;
- hard-gate fields are computed but omitted from promotion;
- undefined metrics default to safe-looking zero/True;
- scientific FAIL returns an execution PASS status without carrying the verdict;
- outputs contain only a thin summary while required tables/reasons/fingerprints are missing.

Tests should assert the exact cell/key set and required output inventory, not merely “both windows appeared somewhere” or “an archive exists.”

## 5. Fail-closed numerical and serialization probes

External-data paths need explicit tests for:

- NaN/Infinity in arrays, metrics, JSON, and dry-run estimates;
- wrong row count/component count or malformed reducer outputs;
- all-undefined correlations/directions;
- rates/fractions outside `[0,1]`;
- input/output path aliasing, symlinks, traversal, and JSON injection;
- transient retry vs permanent/auth/schema failure.

Undefined required evidence must remain undefined and block promotion; it must not become `0`, `True`, or an ordinary zero-vector state.

## 6. Acceptance wording

Use precise boundaries:

- `local synthetic behavior accepted` only after production-shaped fixtures and entry-point tests pass;
- `live contract accepted` only after real schema/artifact preflight or target-side dry-run proves assumptions;
- `scientific result accepted` only after approved real-data execution and analysis.

Do not let “315 tests pass” collapse those three claims into one.
